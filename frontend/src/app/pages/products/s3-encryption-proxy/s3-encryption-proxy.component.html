S3 Encryption Proxy

Transparenter Go-Proxy zwischen beliebigen S3-Clients und S3-kompatiblen Backends. Jedes Objekt wird per Envelope Encryption mit einem eindeutigen Data-Encryption-Key (DEK) verschlüsselt, bevor es den Storage-Layer erreicht — weder der Anbieter noch ein kompromittierter Bucket-Key legt den gesamten Datenbestand offen.

Das Problem

Provider-managed SSE — ob AWS SSE-S3, SSE-KMS oder MinIO-seitige Verschlüsselung — legt die Schlüsselhoheit in die Hände des Anbieters. Compliance-Anforderungen in regulierten Branchen verlangen jedoch Customer-Managed Keys, die niemals die eigene Kontrolldomäne verlassen. Heterogene S3-Backends unterscheiden sich erheblich in ihren Encryption-Features, was eine einheitliche Strategie erschwert. Und Anwendungscode anzupassen oder auf andere SDKs zu wechseln ist in produktiven Umgebungen keine realistische Option.

So funktioniert der Proxy

1

Client spricht Standard-S3

AWS CLI, rclone, Terraform S3 Backend, jedes S3-SDK — keinerlei Code-Änderungen erforderlich. Der Proxy ist ein transparenter Drop-in.

2

Proxy validiert & verschlüsselt

AWS Signature V4 wird geprüft. Pro Objekt erzeugt der Proxy einen eindeutigen DEK, verschlüsselt ihn mit dem KEK (Envelope Encryption) und signiert die Anfrage für das Backend neu.

3

Backend erhält nur Chiffretext

AWS S3, MinIO, Ceph oder jedes andere S3-kompatible Ziel speichert ausschließlich verschlüsselte Daten — Klartext verlässt nie Ihre Kontrolldomäne.

Encryption-Provider

Empfohlen
🔑

RSA Envelope

Ein asymmetrischer RSA-KEK schützt die pro-Objekt generierten AES-Schlüssel. Der Proxy wählt automatisch zwischen AES-GCM für Single-Part-Uploads und AES-CTR für Multipart-Uploads — optimiert für Korrektheit und Streaming-Effizienz. Geeignet für Umgebungen mit hohen Sicherheitsanforderungen.

🗝️

AES Envelope

Ein symmetrischer Master-Key schützt die pro-Objekt generierten AES-Schlüssel. Einfacheres Schlüssel-Management und schnellere Kryptografieoperationen — geeignet für niedrigere Bedrohungsmodelle oder interne Deployments ohne externe Schlüsselinfrastruktur.

Beide Provider können gleichzeitig betrieben werden. Lese-Operationen erkennen automatisch, mit welchem Provider ein Objekt geschrieben wurde — Migration ohne Downtime, kein Maintenance-Window erforderlich.

Funktionsumfang

🔄

Transparente S3-Kompatibilität

Clients sprechen Standard-S3 ohne Anpassungen. Der Proxy fängt Anfragen ab, verschlüsselt und signiert sie neu — Zero Code Changes.

🧩

Per-Objekt-DEKs

Jedes Objekt erhält einen eindeutigen Data-Encryption-Key. Ein kompromittierter DEK legt nur ein einzelnes Objekt offen — der Blast Radius bleibt minimal.

📦

Streaming-Multipart-Uploads

Konfigurierbare Buffer-Größen (4 KB – 2 MB, Standard 64 KB) und Segment-Größen (5 MB – 5 GB, Standard 12 MB) für große Dateien ohne vollständiges In-Memory-Buffering.

🛡️

HMAC-SHA256-Integritätsprüfung

Vier Modi: off, lax (Warnung), strict (Ablehnung) und hybrid (gemischte Policy) — Tamper-Detection ohne Kompromisse.

🔐

AWS Signature V4 Validierung

Inbound-Authentifizierung mit Rate-Limiting, Tracking fehlgeschlagener Versuche und IP-Blocking sowie Clock-Skew-Validierung (Standard: 5 Minuten).

📊

Prometheus-Metriken

Metriken auf Port 9090 unter /metrics — direkt in bestehende Observability-Stacks integrierbar.

⚙️

YAML-Konfiguration mit Env-Var-Substitution

Variablen via ${VAR_NAME} — fehlende Pflicht-Variablen stoppen den Start mit einer klaren Fehlermeldung. Accidental-Plaintext-Betrieb wird so verhindert.

🔀

Zero-Downtime-Provider-Migration

Neues und altes Verschlüsselungsschema können parallel betrieben werden. Lese-Operationen erkennen den verwendeten Provider automatisch — kein Maintenance-Window für Schlüssel-Transitionen.

Abgrenzung zu bestehenden Lösungen

vs. AWS SSE-S3 / SSE-KMS

Bei AWS SSE verwaltet AWS die Schlüssel — Ihre Daten liegen potenziell im Einflussbereich des Anbieters. Der S3 Encryption Proxy hält Schlüssel vollständig in Ihrer Infrastruktur und funktioniert mit jedem S3-kompatiblen Backend.

vs. AWS SSE-C

SSE-C erfordert, dass die Anwendung pro Request Schlüsselmaterial mitschickt. Der Proxy übernimmt die Schlüsselinjektion transparent — die Anwendung überträgt kein Schlüsselmaterial und benötigt keine Anpassungen.

vs. MinIO Server-Side Encryption

MinIO SSE bindet Verschlüsselungslogik an MinIO-spezifische Mechanismen. Der S3 Encryption Proxy ist vollständig anbieterunabhängig und portierbar über beliebige S3-kompatible Backends hinweg.

vs. Client-seitige SDK-Verschlüsselung

Tools wie das AWS Encryption SDK erfordern Code-Änderungen in jeder Anwendung. Der Proxy verschlüsselt automatisch für alle Clients — AWS CLI, rclone, Terraform und jedes S3-SDK profitieren ohne eine Zeile Code-Änderung.
